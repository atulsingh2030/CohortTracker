# PLANNING SKILL

## When to use
When the task involves strategy, architecture, roadmapping, or any thinking-before-doing.
Keywords: plan, roadmap, design, architect, strategy, approach, figure out.

## Inputs required
- Objective (what needs to be achieved)
- Constraints (time, tech stack, team size, known blockers)

## Steps
1. Restate the objective in one sentence to confirm understanding
2. List known constraints and unknowns
3. Break objective into discrete sub-tasks
4. Order sub-tasks by dependency (what must happen before what)
5. Identify risks per sub-task
6. Produce the final structured plan
7. Validate: does the plan cover the objective fully with no orphaned steps?

## Output format
- Objective (1 sentence)
- Constraints
- Ordered task list with dependencies noted
- Risk log (task → risk → mitigation)

## Validation checklist
- [ ] Objective restated correctly
- [ ] No task is floating without a dependency anchor
- [ ] Each risk has a named mitigation
- [ ] Plan is reviewable by a human before execution begins
